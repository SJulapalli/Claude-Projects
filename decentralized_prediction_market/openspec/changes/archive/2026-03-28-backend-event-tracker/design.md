## Context

The PredictionMarket contract is deployed on Sepolia and emits five events: `MarketCreated`, `BetPlaced`, `MarketResolved`, `MarketCancelled`, and `Withdrawal`. Currently there is no off-chain layer — any frontend would need to scan the chain directly for data. This design introduces a lightweight Express.js server that indexes these events into SQLite and exposes read-only REST endpoints.

## Goals / Non-Goals

**Goals:**
- Index all PredictionMarket contract events into a local SQLite database
- Backfill historical events on startup so the database reflects full contract history
- Expose read-only REST endpoints for market listing, market details, per-user bets, and resolution state
- Keep the data model as close to the on-chain representation as possible — no derived or transformed fields

**Non-Goals:**
- Writing to or interacting with the contract (no transactions)
- Authentication, authorization, or rate limiting
- Frontend code or UI
- Real-time WebSocket push to clients (polling via REST is sufficient)
- Supporting multiple contracts or networks simultaneously
- Caching layers or performance optimization beyond SQLite indexes

## Decisions

### Database: SQLite via better-sqlite3
**Decision:** Use SQLite with the `better-sqlite3` npm package for storage.
**Rationale:** The data volume is small (event-driven, Sepolia testnet). SQLite is zero-config, embedded, and requires no separate database process. `better-sqlite3` is synchronous and faster than the callback-based `sqlite3` package. If scaling is needed later, migrating to PostgreSQL is straightforward since the schema is simple.
**Alternative considered:** PostgreSQL — overkill for a testnet indexer with low write volume and single-server deployment.

### Event backfill strategy
**Decision:** On startup, query all historical events from block 0 (or a configurable start block) to the current block, insert them into the database (skipping duplicates), then subscribe to new events via `contract.on()`.
**Rationale:** This ensures the database is complete even if the server was offline when events were emitted. Using a `last_synced_block` tracking table avoids re-scanning the entire history on every restart.
**Alternative considered:** Only listen for live events — simpler but the database would be incomplete if the server restarts.

### Database schema
**Decision:** Three tables mirroring the contract's data model:
1. `markets` — one row per market (id, creator, question, outcomes as JSON array, betting_deadline, resolution_deadline, resolved, cancelled, winning_outcome, total_pool). Updated by MarketCreated, MarketResolved, MarketCancelled events.
2. `bets` — one row per bet event (market_id, bettor, outcome_index, amount, tx_hash). Append-only from BetPlaced events. A user can bet multiple times, so there may be multiple rows per (market_id, bettor, outcome_index).
3. `withdrawals` — one row per withdrawal event (market_id, user, amount, tx_hash). Append-only from Withdrawal events.

**Rationale:** Directly mirrors the contract storage. `total_pool` on markets is updated by summing from BetPlaced events. Outcomes are stored as a JSON string since SQLite doesn't have array types, and the contract stores them as a string array.

### API structure
**Decision:** RESTful endpoints under `/api/`:
- `GET /api/markets` — list all markets
- `GET /api/markets/:id` — full details for a specific market (including outcomes, bets summary)
- `GET /api/markets/:id/resolution` — resolution state of a market
- `GET /api/users/:address/bets` — all markets a user has bet on with their bet details

**Rationale:** Simple resource-based REST. No pagination needed at testnet scale. Each endpoint maps directly to one of the user's stated query needs.

### Contract ABI source
**Decision:** Import the ABI from the Hardhat compilation artifacts (`artifacts/contracts/PredictionMarket.sol/PredictionMarket.json`).
**Rationale:** The ABI is already generated by Hardhat compilation. Importing it directly avoids maintaining a separate copy that could drift.

### Server directory structure
**Decision:** All server code lives in a `server/` directory at the project root, with its own `package.json` for backend-specific dependencies.
**Rationale:** Keeps the server isolated from the Hardhat project. The server has different dependencies (express, better-sqlite3) than the contract tooling. A shared root `package.json` would mix concerns.

## Risks / Trade-offs

- **Event ordering** → SQLite inserts are synchronous via better-sqlite3, so events are processed in order. If the RPC returns events out of order during backfill, the data is still correct since each event is idempotent (keyed by tx_hash or upserted by market_id).
- **RPC reliability** → If the Sepolia RPC drops the WebSocket/polling connection, events could be missed. Mitigation: On reconnect, re-backfill from `last_synced_block` to catch up.
- **No real-time push** → Frontends must poll. Acceptable for testnet; WebSocket support could be added later.
- **SQLite concurrency** → SQLite handles one writer at a time. Since only the event listener writes and volume is low, this is not a concern for testnet scale.
